# 代码规范

## 目标

代码首先要清楚，其次才是技巧。

项目代码应优先服务阅读、修改和定位问题，而不是展示写法。

## 核心要求

- 命名要直接表达职责
- 结构要反映真实边界
- 函数和组件只做一层明确的事情
- 避免为了抽象而抽象
- 避免为了灵活而引入隐藏复杂度

## 可读性

- 优先写自解释代码
- 优先让调用关系清楚
- 优先让数据流清楚
- 避免依赖阅读者自行猜测上下文

## 注释

注释不是默认配置。

只有在以下情况才写注释。

- 代码意图无法从结构和命名直接看出
- 存在外部约束
- 存在容易误解的边界条件
- 存在不得不保留的实现取舍

不写解释显而易见行为的注释。

不写把代码再重复一遍的注释。

## 结构

- 模块边界要稳定
- 文件职责要单一
- 公共抽象要克制
- 状态管理要明确来源和生命周期
- 默认交互路径要尽量短
- 非必要配置不要进入主路径
- 高级能力应与默认能力分层组织

## 修改原则

- 先理解现有结构，再修改
- 优先局部修正，避免无必要扩散
- 新增逻辑时优先复用已有边界
- 保持接口、命名和行为一致
- 不把内部实现缺陷转化为用户必须补充的配置步骤
- 默认行为应服务最简单的使用路径
- 面向 AI 的输入格式和面向程序接口的输入格式要明确区分

## 质量要求

- 代码要干净
- 错误路径要明确
- 输出和副作用要可追踪
- 能删掉的复杂度不要留下
