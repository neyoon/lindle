# 开发细节

## 范围

这份文档记录本项目当前明确执行的开发要求。

当规范发生变化时，先更新这里，再推进对应改动。

## 文档组织

- docs 目录按主题分层，不再堆在根下
- 架构文档统一放在 architecture
- 开发规范统一放在 development
- 版本相关文档统一放在 versioning

## README 要求

README 只保留项目入口信息。

README 负责说明项目是什么、怎么启动、去哪里继续阅读。

README 不承担完整架构说明，不承担详细开发规则。

## 架构文档要求

- 从项目定位开始
- 先说明系统边界
- 再说明核心对象
- 再说明运行路径
- 最后落到设计原则

架构文档要保持简洁，不为了完整而牺牲主线。

## 文档写作要求

- 简洁
- 明白
- 不说废话
- 自顶向下
- 关注架构
- 不写代码
- 不写示例代码

文档里不放代码片段，不放演示性实现，不把文档写成教程型代码说明。

## 代码写作要求

- 代码保持干净
- 代码优先自解释
- 命名优先表达职责
- 结构优先表达边界
- 不过度注释

只有在意图、约束或边界不够明显时才补注释。

## 默认路径要求

默认路径必须优先服务用户最简单的使用方式。

如果系统内部存在多种数据表示，不应把这种差异直接暴露给用户。

非必要的选择不要放进主路径。

确实有必要但低频的配置，应该通过默认隐藏的高级选项提供。

尤其在 Flow 中：

- AI 节点默认可以接收适合模型理解的文本格式
- Plugin 节点默认应接收适合程序执行的结构化格式

当内部语义不一致时，优先调整系统默认行为，而不是要求用户增加手工配置。

显式模板和高级配置属于补充能力，不应成为跑通默认流程的前提。

## 测试要求

测试优先验证系统在真实流程中的反应。

不要把大量精力消耗在没有业务价值的格式校验上。

简单的输入输出格式验证仍然需要保留，但应保持最小覆盖。

测试重点应放在以下问题上。

- 用户按主路径操作时，系统是否给出符合预期的反应
- 默认行为是否真的足够简单
- 多阶段流程中，状态切换和结果传递是否正确
- 工具、Flow、Agent 串起来时，是否能稳定完成完整任务

## 版本管理要求

每个版本单独建目录。

版本目录至少记录以下内容。

- 版本目标
- 本次范围
- 准备修改的事项
- 已完成事项
- 后续待跟进事项

较大的文档调整或结构调整，应先落到对应版本文档，再实施。
