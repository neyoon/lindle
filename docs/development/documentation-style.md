# 文档规范

## 目标

文档用于解释系统，不用于展示作者。

写文档时要帮助读者更快建立结构化理解，而不是堆积信息。

## 写法要求

- 简洁
- 明白
- 自顶向下
- 保持关注架构
- 不说废话

## 组织方式

每份文档都应先回答三个问题。

- 这份文档要说明什么
- 这件事在系统中的位置是什么
- 读完之后读者应该知道什么

先讲整体，再讲局部。

先讲边界，再讲细节。

先讲关系，再讲实现。

## 内容边界

文档只写文档。

不要在文档里放代码。

不要在文档里放示例代码。

不要把实现细节大段复制进文档。

不要用文档替代代码阅读。

## 表达要求

- 句子要短
- 段落要收敛
- 一个部分只讲一件事
- 避免营销语言
- 避免重复定义同一个概念

## 架构文档要求

架构文档优先解释系统边界、核心对象、运行路径和设计原则。

架构文档不追求面面俱到，追求主线清楚。

## 维护要求

- 文档结构变化要同步更新总览
- 版本相关内容要进入 versioning
- 开发要求变化要进入 development-details
