# 测试文档

## 目标

测试的目标不是证明每个字段都能被校验一遍。

测试的目标是验证系统在真实使用流程中，是否给出符合预期的反应。

## 基本原则

- 先测主路径
- 先测完整流程
- 先测用户感知明显的反应
- 格式校验保留最小覆盖
- 不用低价值用例淹没高价值问题

## 测试重点

测试应优先回答以下问题。

- 用户按默认路径操作时，系统能不能顺利完成任务
- 当系统进入多阶段执行时，阶段切换是否清楚
- 当 Agent 调工具、工具返回、Agent 再继续回复时，表现是否符合预期
- 当 Flow 经过 AI、Plugin、Output 多个节点时，数据传递是否稳定
- 默认行为是否真的比手工配置更简单

## 低优先级测试

以下内容需要测试，但只需要做最小覆盖。

- 空输入
- 缺失字段
- 字段类型错误
- 简单的输入输出格式校验

这些测试的价值在于防止明显回归，不在于构成测试主体。

## 推荐测试结构

测试应按场景组织，而不是按字段组织。

每个测试场景至少包含四部分。

- 前提条件
- 用户动作
- 预期反应
- 失败信号

## 核心测试场景

### Agent 流式返回

重点不是验证某个事件字段名字对不对。

重点是验证用户看到的流式过程是否符合预期。

应重点关注：

- 开始回复前是否进入思考状态
- 调工具前后是否有明确状态变化
- 工具执行中是否持续反馈进度
- 最终回复是否在工具完成后继续生成
- reasoning 是否只出现在合理位置，不重复，不混乱

### Flow 到 Plugin 的数据传递

重点不是验证某个 JSON 字段是否逐字相等。

重点是验证默认路径是否足够可靠。

应重点关注：

- 单一上游结果直接进入 Plugin 时是否能跑通
- 多个上游结果进入 Plugin 时是否有稳定行为
- 在需要字段转换时，显式模板是否能解决问题
- 默认路径是否不要求用户额外补配置

### 自动生成 Flow

重点不是模型生成了多少字符。

重点是生成结果能不能执行。

应重点关注：

- 生成的 Flow 在默认主路径下是否能保存并运行
- 遇到 Plugin 时是否正确使用默认输入语义
- 只有在确实需要时才生成额外模板
- 生成结果是否保持结构清楚，不过度暴露高级选项

### Agent 对话保存

重点不是把每条消息都落盘一次就算完成。

重点是验证用户重新进入 Agent 时，是否还能继续上一次对话，而不是重新开始。

应重点关注：

- 用户发送消息并得到完整回复后，对话是否被保存
- tool_call 和 tool_result 是否一起保存，避免链路断裂
- 刷新页面或重新进入 Agent 后，历史对话是否能恢复
- 主动清空对话后，后续请求是否从新的上下文开始

## 通过标准

一个测试场景通过，不代表字段全部覆盖。

一个测试场景通过，代表用户按这个流程走时，系统反应正确，主要结果正确，状态切换合理。

## 失败处理

如果失败来自流程断裂、状态错误、默认路径不可用，这类问题优先级最高。

如果失败只来自低价值格式差异，但不影响主路径使用，优先级应低于流程问题。
