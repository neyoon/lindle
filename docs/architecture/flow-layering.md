# Flow 三层架构

## 目标

Flow 的创建、编辑和执行不应共用同一份结构。

如果把“用户意图”“编辑器结构”“执行结构”揉在一起，系统会逐步出现这些问题：

- AI 被迫直接生成最终执行细节
- 用户重命名后容易破坏引用
- 编辑器结构和执行引擎互相绑死
- 系统隐藏了太多语义变化，导致排错困难

因此 Flow 需要明确分成三层：

- FlowSpec
- CanonicalFlow
- ExecutionPlan

## 总体关系

### 三层职责

- FlowSpec：表达用户真正操作的任务结构
- CanonicalFlow：表达系统内部的稳定编辑结构
- ExecutionPlan：表达运行时执行结构

### 主链路

系统的主链路应当是：

- 用户需求
- AI 生成或修改 FlowSpec
- 系统把 FlowSpec 收敛成 CanonicalFlow
- 前端编辑器展示和编辑 FlowSpec 语义
- 系统在保存前或运行前校验 CanonicalFlow
- 系统把 CanonicalFlow 编译成 ExecutionPlan
- 引擎执行 ExecutionPlan

### 用户可见性

- 用户应直接接触 FlowSpec 表达的主步骤结构
- 用户不需要直接编辑 CanonicalFlow 细节
- 用户不直接接触 ExecutionPlan

编辑器中的 Flow 应首先表达 FlowSpec，而不是暴露 CanonicalFlow 的内部组织方式。

## FlowSpec

### 定位

FlowSpec 是用户主操作层，也是 AI 的主创建层。

它负责表达用户想做什么，不负责表达系统最终怎么执行。

### 适用场景

- AI 根据需求创建新 Flow
- AI 根据指令修改已有 Flow 的主结构
- 用户在编辑器中查看、调整和重排主步骤
- 系统先把自然语言需求整理成主步骤结构

### 应包含的信息

FlowSpec 至少应包含：

- 流程目标
- 输入定义
- 主步骤列表
- 步骤依赖关系
- 最终输出定义

### 最小字段建议

顶层建议至少包含：

- `goal`
- `inputs`
- `outputs`
- `steps`

每个步骤建议至少包含：

- `step_ref`：稳定步骤引用
- `title`：显示标题
- `type`：`collect | process | tool | result`
- `purpose`：该步骤的主目标
- `depends_on`
- `tool_id`：仅工具步骤需要
- `output_contract`

### 不应包含的信息

FlowSpec 不应直接承担这些细节：

- `column_id`
- `block_id`
- 最终 `connections`
- 插件输入模板字符串
- AI prompt 拼接细节
- 运行时隐藏适配逻辑
- 前端展示状态

这些内容属于收敛层或执行层，不应成为 AI 和用户在创建阶段的默认负担。

### FlowSpec 的引用语义

FlowSpec 默认只允许使用稳定引用，不允许按显示名引用。

- 输入引用使用 `{{inputs.<input_ref>}}`
- 步骤完整输出使用 `{{steps.<step_ref>}}`
- 步骤字段输出使用 `{{steps.<step_ref>.<key>}}`

这条规则必须贯穿 AI 生成、前端编辑和运行时解析。

## CanonicalFlow

### 定位

CanonicalFlow 是系统内部标准编辑结构。

它不再承担“用户默认直接编辑”的职责，而是承接保存、校验、对比、导出和继续编译。

### 适用场景

- 持久化保存
- 保存前和运行前校验
- AI 修改后的差异比较
- 导出和版本对比

### 应包含的信息

CanonicalFlow 应包含：

- 稳定的 workflow 标识
- 名称和描述
- 标准化后的列结构
- 标准化后的块结构
- 稳定的字段引用标识
- 明确的连接和依赖关系
- 用户显式配置过的关键设置
- 可校验的完整结构

### 最小字段建议

顶层建议至少包含：

- `workflow_id`
- `name`
- `description`
- `inputs`
- `columns`
- `validation`
- `meta`

每个块建议至少包含：

- `block_id`
- `block_ref`
- `name`
- `type`
- `config`
- `input_bindings`
- `output_schema`
- `connections`

输入字段建议至少包含：

- `field_id`
- `field_ref`
- `label`
- `field_type`
- `required`
- `default`

### CanonicalFlow 的关键要求

- 显示名称不承担内部引用语义
- `block_ref` 与 `name` 分离
- `field_ref` 与 `label` 分离
- 所有依赖关系都可校验
- 所有默认值都可补齐
- 所有块的输入输出语义都明确

### 手动连线的定位

手动连线是 CanonicalFlow 中默认具备的核心控制能力。

它的作用是让用户在需要时显式控制输入流向，而不是被系统默认推导结果完全锁死。

但手动连线不应成为默认必需路径。

系统默认仍应先给出可工作的数据流，用户只在这些场景下再显式连线：

- 覆盖默认上游来源
- 缩小输入范围
- 指定多路来源中的某一路
- 处理系统无法确定的 plugin 输入来源
- 让某个块只消费特定字段或特定分支结果

因此更准确的原则是：

- 手动连线默认存在
- 自动数据流默认先工作
- 显式连线用于覆盖默认行为

## ExecutionPlan

### 定位

ExecutionPlan 是运行时结构。

它不是给用户编辑的，也不是给 AI 创建的，而是给执行引擎消费的。

### 为什么必须存在

CanonicalFlow 优化的是“可编辑、可保存、可校验”。

ExecutionPlan 优化的是“可解析、可调度、可运行”。

这两个目标不相同，因此不应使用同一层结构承担。

### 适用场景

- 运行前编译
- 依赖展开
- 默认数据流推导
- 插件输入生成
- 隐式适配注入
- 引擎执行

### 应包含的信息

ExecutionPlan 应包含：

- 已展开的执行步骤
- 已解析的依赖图
- 每一步的实际输入来源
- 系统自动补入的适配动作
- 运行时默认值
- 引擎真正需要的执行顺序

### 最小字段建议

顶层建议至少包含：

- `plan_id`
- `workflow_id`
- `steps`
- `edges`
- `compiled_at`

每个执行步骤建议至少包含：

- `step_id`
- `from_block_ref`
- `runtime_type`
- `resolved_inputs`
- `resolved_outputs`
- `execution_policy`
- `system_adapters`

### 与 CanonicalFlow 的区别

- CanonicalFlow 面向系统内部编辑结构
- ExecutionPlan 面向引擎

CanonicalFlow 中允许保留系统整理后的稳定列结构、块结构和连接细节，但这些不是用户主操作语义。

ExecutionPlan 中则应只保留引擎需要的确定性运行信息。

## 三层之间的转换

### FlowSpec -> CanonicalFlow

系统应自动完成这些工作：

- 生成稳定标识
- 补齐默认值
- 规范步骤顺序
- 将高层步骤映射到列和块
- 生成可校验的连接信息
- 为输入字段生成稳定引用

### CanonicalFlow -> ExecutionPlan

系统应自动完成这些工作：

- 解析依赖关系
- 展开隐式默认数据流
- 保留用户手动连线的优先级
- 判断哪些适配逻辑允许隐藏
- 生成插件实际输入
- 生成引擎可执行的运行顺序

## 块职责边界

### 总规则

只有“不会改变用户理解的语义，且系统能确定性完成”的工作，才能隐藏在系统里。

否则必须显式成块。

### 三种判定结果

对于任一步骤或中间动作，系统最终只允许三种判定：

- `must_split`：必须显式成块
- `may_hide`：允许下沉为系统内部适配
- `must_surface`：必须显式暴露给用户，不能藏在 Output 或默认行为里

### 必须拆块的情况

满足任意一条时，应判定为 `must_split`：

- 它是用户可理解的独立任务目标
- 它的结果值得单独查看、调试或复用
- 它改变了任务语义，而不只是格式
- 它构成明显的错误边界
- 它被多个下游共同依赖
- 它需要单独配置模型、插件或运行策略

### 可以隐藏的情况

同时满足这些条件时，可判定为 `may_hide`：

- 不改变任务语义
- 只是数据适配或格式整理
- 系统可以确定性完成
- 不需要用户单独配置
- 用户通常不需要单独查看这一层结果

### 必须显式暴露的情况

满足任意一条时，应判定为 `must_surface`：

- 多来源汇总
- 有损转换
- 过滤、裁剪、选择
- 评分、排序、决策
- 人类可感知的文本重写
- 最终展示文案整理

这类职责不能被 Output 隐式承担。

## 步骤语义分类

系统在判断拆块时，不应直接让大模型决定“拆还是不拆”。

更稳妥的做法是：

- AI 先理解步骤意图
- 系统再做规则化裁决

为此，每个步骤建议引入内部语义分类：

- `semantic_work`
- `deterministic_mapping`
- `aggregation`
- `presentation`
- `transport`

### 各类默认处理

- `semantic_work`：默认独立成块
- `deterministic_mapping`：默认隐藏到系统适配层
- `aggregation`：默认独立成块
- `presentation`：默认独立成块
- `transport`：默认隐藏

## AI 与系统的职责分工

### AI 负责

- 识别主任务步骤
- 识别每个步骤的目标
- 给出粗粒度依赖关系
- 对步骤做语义分类建议

### 系统负责

- 决定哪些步骤必须拆块
- 决定哪些映射逻辑可以隐藏
- 决定何时生成显式汇总块
- 决定何时生成显式展示块
- 保证最终结构稳定、可校验、可执行

AI 可以参与理解，但不能单独决定最终结构边界。

## Output 的职责

Output 的职责应收口为：

- 标记流程终点
- 暴露最终结果
- 保留最终结构化输出

Output 默认不负责：

- 文本重写
- 结果摘要
- 展示文案整理
- 面向用户的最终可读文案生成

如果需要这些能力，应使用显式的展示类块，例如：

- formatter
- summary
- presenter

## 典型例子

### 必须拆开的例子

- 写文章 -> 润色文章
- 获取数据 -> 分析数据 -> 生成图表
- 提取候选项 -> 排序 -> 输出最终选择

### 可隐藏的例子

- 上游字段改名
- 单层结构重排
- 为 plugin 补常量字段
- 默认数据流透传

### 取决于系统是否可确定性完成的例子

- 分析结果 -> 转成 plugin 所需字段

如果只是确定性的字段映射，应隐藏。

如果包含业务解释、筛选、语义改写或显著的信息损失，应拆成显式块。

## 设计原则

- 创建、编辑、执行三层分离
- AI 负责表达主结构，系统负责收敛和裁决
- 稳定引用不依赖显示名称
- 默认路径优先简单，但不能隐藏关键语义变化
- 对用户重要的边界必须显式存在
