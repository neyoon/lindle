# 架构文档

## 项目定位

Lindle 是一个面向 AI 能力编排的工作台。

它解决的不是模型训练问题，而是能力组织问题。项目的目标是把输入、模型处理、工具调用和结果输出组织成可理解、可复用、可扩展的结构。

Lindle 的底层原则是上手简单，结构清晰。用户应先通过直观方式创建和组织能力；系统再在内部把这些能力整理成可执行、可调度、可复用的结构。

## 名称

Lindle 取自 `Loom` 和 `Spindle` 这两个方向。

这个命名表达的是把输入、模型、工具和结果编织成结构，再由系统驱动执行。

它强调的是“能力编排”和“结构收敛”，而不是单一聊天入口。

## 系统边界

系统包含四个核心面向。

- Flow：可视化能力编排
- Agent：面向对话的能力调度
- Skill：Agent 可调用的能力关联层
- Plugin：Flow 内部可执行的节点能力

Flow 和 Agent 是两个入口。Skill 和 Plugin 是两种执行能力的挂载方式。它们最终共同服务于统一的能力编排。

用户的默认路径应当是从 Flow 开始，而不是从 Tool、Plugin、Schema 或 function calling 开始。用户创建 Flow，系统把 Flow 沉淀为可执行能力；用户选择多个 Flow 时，系统可以自动生成 Skill，让 Agent 通过这个 Skill 使用这一组能力。

## 顶层结构

系统分为四层。

- 前端层：负责编辑、配置、对话和结果展示
- 接口层：负责暴露工作流、执行、插件、设置和 Agent 能力
- 引擎层：负责 Flow 执行和 Agent 调度
- 存储与模型层：负责用户隔离存储和外部模型访问

这个分层的重点是把界面、接口、执行和基础设施分开，减少耦合。

## 核心对象

### Tool

Tool 是统一执行抽象。

只要某个能力最终可以被执行，它在系统视角里就属于 Tool。

Tool 可以是原子工具，也可以是由多个步骤组成的组合工具。Flow 在 Agent 侧被使用时，本质上就是一个结构化的组合 Tool。

### Flow

Flow 是可视化工作流，也是用户制作能力的主要方式。

Flow 由多个 Column 组成。Column 按顺序执行。同一 Column 内的多个 Block 并行执行。Block 是最小执行单元。

Flow 有两种角色。

- 作为独立流程直接执行
- 作为可被 Skill 暴露给 Agent 的组合 Tool

Flow 本身不直接承担 Agent 侧的 Skill 语义。Flow 负责表达能力如何执行，Skill 负责表达这些能力如何被 Agent 理解、选择和调用。

### Agent

Agent 是对话式调度器。

Agent 自身不承担所有能力实现。它通过模型理解用户意图，再结合可用 Skill 决定是否调用工具、以什么顺序调用工具，以及如何组织最终回复。

### Skill

Skill 是 Agent 使用的能力关联层。

Skill 分两类。

- 管理型 Skill：负责管理并暴露多个 Flow
- 独立型 Skill：自身就是完整能力

在 Lindle 中，Skill 的重点不是把每个工具单独包装成函数，而是把一组能力整理成 Agent 可调用的接口。对于由 Flow 组成的能力集合，真正的 Skill 是类似 Flow 执行器这样的关联层，而不是某一个 Flow 本身。

Skill 至少应承担这些职责：

- 描述这一组能力适合解决什么问题
- 暴露这一组能力中可用的 Flow 或 Tool
- 帮助 Agent 在多个 Flow 之间选择合适目标
- 将用户意图和对话上下文适配成 Flow 需要的输入
- 将 Flow 的执行结果整理成 Agent 可以继续推理和回复的内容

因此，多选 Flow 并导出为 Skill 不是简单的导出动作，而是把多个组合 Tool 组织成 Agent 能力接口的过程。

### Plugin

Plugin 是 Flow 内部节点能力。

Plugin 运行在 Flow 中，不直接面向 Agent。Skill 运行在 Agent 中，不直接作为 Flow 内部节点。两者职责分开。

Plugin 的默认输入语义应当面向程序接口，而不是面向自然语言阅读。

这意味着 Plugin 默认应接收结构化上游结果，而不是带说明性包装的文本。

如果需要做格式转换，应由系统默认行为或显式模板承担，不应默认要求用户额外补充转换步骤。

## 运行路径

### Flow 路径

用户在前端配置 Flow。

前端编辑的 Flow 应首先表达 FlowSpec 主步骤。

AI 创建和修改 Flow 时，系统先生成或修改 FlowSpec，再收敛成 CanonicalFlow。

保存前和运行前，系统再对 CanonicalFlow 做统一校验，并在执行前编译成 ExecutionPlan。

执行时，Flow Engine 读取 ExecutionPlan，按列顺序推进，同列并行执行，持续维护上下文数据，并在结束时生成统一输出。

Flow 的创建结构与三层收敛定义见 `flow-creation.md` 和 `flow-layering.md`。

### Skill 生成路径

用户在 Flow 列表中选择一个或多个 Flow。

系统根据这些 Flow 的名称、描述、输入字段、输出形式和执行语义生成 Skill。这个 Skill 不应只是保存 Flow ID 列表，还应形成面向 Agent 的能力说明和调用契约。

生成后的 Skill 应表达：

- 这组 Flow 共同构成什么能力
- 每个 Flow 适合什么场景
- 调用时需要哪些输入
- 输出结果应如何解释
- Agent 是否只能访问这组被绑定的 Flow

这个路径的目标是让用户通过简单选择完成能力组织，而不是要求用户理解底层 Tool、Plugin 或 function calling 细节。

### Agent 路径

用户在前端向 Agent 发送消息。

后端根据 Agent 配置构建模型输入和可用 Skill 列表。

模型返回普通回复时，结果直接结束。

模型返回工具调用时，Agent Engine 执行对应 Skill。对于 Flow 执行器类 Skill，Skill 再选择或执行绑定的 Flow，把结果回注到对话上下文，直到得到最终回复或达到轮次上限。

## 关键关系

- Flow 是用户制作能力的结构化载体
- Agent 是能力调度器
- Skill 是 Agent 面向一组能力的关联接口
- Plugin 是 Flow 的执行节点
- Tool 是统一抽象

项目的关键设计不是把所有能力揉在一起，而是让不同入口共享同一套能力组织方式。

核心关系可以概括为：

- Plugin 提供原子能力
- Flow 组织原子能力，形成组合 Tool
- Skill 暴露一组 Flow 或 Tool，形成 Agent 可理解的能力接口
- Agent 根据用户意图调度 Skill

因此，Flow 组织能力，Skill 暴露能力，Agent 调度能力。

## 存储与部署

当前实现仍以文件存储为主，但数据已经按用户隔离。

这说明项目当前阶段强调低复杂度部署，同时已经为多用户场景预留了组织方式。

模型调用使用 OpenAI 兼容接口，重点是接入灵活，而不是绑定单一提供方。

## 设计原则

- Flow 和 Agent 同时存在，但职责明确
- 能力抽象统一，但使用场景区分清楚
- 架构优先保持清晰，再谈扩展
- 创建结构、编辑结构、执行结构分层
- 默认路径应简单，复杂能力通过 Skill 和 Plugin 扩展
- 用户应优先理解 Flow 和 Agent，不应被迫理解 Tool、Plugin、Schema 或 function calling
- Skill 应降低 Agent 使用能力的成本，而不是增加用户配置负担
- 多选 Flow 生成 Skill 是默认的能力组织路径，应优先由系统生成说明、路由和调用契约
- 文档表达应服务架构理解，而不是堆信息
- 用户默认路径优先于内部实现方便
- 新功能设计先看是否增加用户使用成本，再看是否方便内部实现
- AI 节点和 Plugin 节点可以共享编排结构，但不应共享错误的默认输入语义
- 内部数据适配应尽量由系统承担，不应转嫁为用户额外操作
- 非必要的选择不应默认暴露
- 低频但必要的能力应进入默认隐藏的高级选项
