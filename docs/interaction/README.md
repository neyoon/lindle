# 简化交互

## 目标

简化交互文档用于定义 Lindle 如何把复杂系统状态转成用户一眼能理解的界面反应。

它关注的不是视觉风格本身，而是交互语义。

用户不应该先读错误文本，再猜哪里坏了。界面应先用状态、颜色、位置和动作告诉用户：

- 发生了什么
- 影响哪里
- 下一步怎么修

## 位置

简化交互位于架构和视觉语言之间。

- architecture 说明系统是什么
- aesthetic 说明界面长什么样
- interaction 说明界面如何用简单反应帮助用户理解系统

## 原则

- 状态先于文字
- 位置先于解释
- 修复入口先于错误详情
- 视觉反应要稳定，不要每个场景重新发明
- 默认只暴露用户下一步需要知道的信息

## 状态语言

每种状态都应有稳定的视觉语义。

- 正常：结构可继续使用，不需要用户处理
- 运行中：系统正在工作，用户应等待或可选择停止
- 完成：动作成功结束，结果可查看
- 需要补充：还不能完整运行，但问题不严重
- 阻塞错误：必须处理，否则不能继续
- 警告：可以继续，但结果可能不符合预期

颜色只能辅助表达状态，不能成为唯一表达。状态还应通过图标、位置、文案或边框变化确认。

## 交互要求

当系统发现问题时，界面应遵守这个顺序：

1. 在问题所在区域显示状态
2. 在全局入口显示汇总状态
3. 提供直接跳转或打开配置的动作
4. 再提供详细原因

不要只弹出报错。

不要只说“错误在第 3/4 步”。

应该让用户看到对应步骤已经变成明确的问题状态，并且可以直接进入修复。
