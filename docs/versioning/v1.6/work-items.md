# 版本 1.6 工作项

## 本次范围

- 引入 Flow Outline 作为用户默认创建层
- 保持现有 FlowSpec / CanonicalFlow / ExecutionPlan 继续作为系统内部链路
- 不删除 collect / result / tool / process 这些执行结构
- 不把字段配置重新推回默认入口

## 本次原则

- 用户先表达意图结构，系统再生成执行结构
- Flow Outline 面向用户，FlowSpec 面向 AI 和系统，CanonicalFlow 面向编辑，ExecutionPlan 面向执行
- 输入输出块可以从默认用户心智中隐藏，但不能从系统执行结构中消失
- 大纲应尽量简单，不应变成另一套复杂配置表单

## 新增结构

### Flow Outline

Flow Outline 是用户默认创建层。

它表达流程大纲，不表达完整执行细节。

它至少应包含：

- 目标
- 输入材料描述
- 步骤列表
- 期望输出
- 可选约束

步骤只需要表达用户可理解的任务。

例如：

- 收集客户资料
- 分析客户需求
- 生成报价方案
- 输出邮件草稿

这些步骤不需要用户先标记为具体 block 类型。

### FlowSpec

FlowSpec 继续承担系统主步骤结构。

1.6 中，FlowSpec 的主要来源会从“直接 AI 编辑 Workflow”扩展为“从 Outline 生成”。

FlowSpec 负责把大纲转成：

- 输入字段
- 主步骤
- 依赖关系
- 工具候选
- 输出定义

### Workflow

Workflow 仍然是当前前端和保存链路兼容的结构。

1.6 不直接推翻它。

用户进入高级编辑时，仍然看到现有画布。

### ExecutionPlan

ExecutionPlan 仍然只面向执行引擎。

用户不需要直接感知它。

## 准备修改

### 1. 大纲创建入口

新增创建 Flow 的默认入口。

用户可以输入：

- 一句话目标
- 多行步骤
- 期望输出

系统生成 Flow Outline，再生成 FlowSpec。

高级入口仍然保留，允许用户直接进入现有画布。

### 2. Outline 到 FlowSpec

新增转换链路。

转换时系统负责推断：

- 是否需要输入
- 输入是什么材料
- 哪些步骤可以并行
- 哪些步骤必须串行
- 哪些步骤适合调用工具
- 是否需要结果步骤

推断结果可以被用户在高级编辑中修正。

### 3. 输入输出默认隐藏

输入和输出仍然存在，但默认不作为用户必须配置的步骤暴露。

默认逻辑：

- 用户写输入材料描述
- 系统生成 collect 结构
- 用户写期望输出
- 系统生成 result 结构
- 字段和输出 key 进入高级选项

这样既保留执行结构，又降低上手成本。

### 4. 大纲视图和画布视图的关系

需要明确两种视图的边界。

大纲视图：

- 编辑意图结构
- 适合新建和早期调整
- 不显示所有执行细节

画布视图：

- 编辑执行结构
- 适合调试、工具映射和高级配置
- 显示块、连接、字段和输出结构

两者不是互相替代，而是不同层级。

### 5. 避免大纲覆盖高级配置

从大纲重新生成执行结构时，不能简单重建整个 Flow。

需要保护：

- 已配置的工具映射
- 手动连接
- 模型选择
- 输出结构
- 用户手动调整过的提示词

如果无法安全合并，应要求用户确认。

### 6. 和 1.5 健康检查衔接

Outline 生成的 Flow 应直接进入健康检查。

健康检查负责告诉用户：

- 哪些结构已经准备好
- 哪些地方需要补充
- 哪些地方可以稍后高级配置

这样用户不会从大纲直接掉进复杂配置。

## 不在本次范围

- 不做完整版本对比系统
- 不做自动重构所有历史 Flow
- 不把 Outline 做成通用项目管理工具
- 不让用户在 Outline 层配置所有字段和工具参数

## 关键设计问题

### 是否还需要 collect 和 result

需要。

但它们应更多作为系统生成的执行结构存在。

用户默认关注输入材料和期望结果，不关注字段与输出块。

### Outline 是否会取代画布

不会。

Outline 是创建层和意图层。

画布是执行层和调试层。

普通用户可以多数时间留在 Outline，复杂场景再进入画布。

### 是否应该放在 1.5

不应该。

1.5 要先把现有 Flow 做稳。

1.6 再引入 Outline，风险更低，也更符合版本节奏。
