# 版本 1.3 工作项

## 本次范围

- 收口 Flow 的用户使用成本问题
- 收口 Flow 的关键概念和默认语义
- 处理 Flow 编辑器中对非技术用户不友好的交互
- 明确块职责边界，避免一个块承担过多隐含责任
- 收口 Flow 创建结构，明确 FlowSpec 与 CanonicalFlow 的内部职责
- 更新架构文档与产品命名

## 本次原则

- 优先降低用户理解成本，而不是优先增加表达能力
- 显示名称不应承担内部引用语义
- 默认路径应尽量不要求用户手写模板或理解 JSON 结构
- 错误应尽量前置，在保存前和运行前暴露
- 块职责应清楚，但不能把结构拆得让 AI 和用户都难以使用

## 准备修改

### 1. 引用标识稳定化

- 为 Block 增加稳定引用标识，不再直接使用显示名称作为模板变量解析依据
- 为 Input 字段增加稳定引用标识，不再因为改 label 自动改引用名
- 重命名时保留原引用，或在系统内部自动迁移引用关系

### 2. 保存前 / 运行前统一校验

- 增加 Flow 校验入口
- 校验重复输入字段、非法块名、无效连接、无效变量引用、缺失配置
- 将错误直接反馈到编辑器，而不是等到运行时报错

### 3. Output 块语义收口

- 默认保留上游原始结构化结果
- 不再把 Output 默认做成“结构化结果转文本”
- 如果需要文本化展示，使用显式能力，而不是让 Output 隐式承担格式转换

### 4. Plugin 配置简化

- 当插件有 input_schema 时，优先提供字段映射式配置
- 让用户通过选择来源值完成配置，而不是默认手写 JSON 模板
- 保留模板输入作为高级能力

### 5. 编辑器交互优化

- 减少 `prompt()` 弹窗式配置
- 让 repeat、重命名、输出 key 编辑变成显式表单
- 让连线和来源关系更容易理解和修改
- 评估未保存草稿直接运行的可行性

### 6. 块职责边界优化

- 梳理哪些块现在承担了过多作用
- 明确哪些职责应该拆开，哪些职责应该保留在同一个块内
- 建立“默认简洁结构”和“必要时展开结构”的规则

## 重点设计问题

### 块是否应该更细粒度

存在这样的问题：当前某些块同时承担“处理内容”“格式转换”“给下游准备结构”多个作用。

这会降低可理解性，也会让排错困难。

但如果把每一步都展开成独立块，用户画布会明显变复杂，AI 生成 Flow 时也更容易产出大量框架性结构。

因此 1.3 需要解决的不是“统一拆细”或“统一不拆”，而是建立明确规则：

- 默认情况下，一个块只承担一个用户可理解的主职责
- 只有在格式转换、结构适配、分支汇总这类职责足够独立时，才显式拆成独立块
- 对 AI 生成来说，默认先生成最小可理解结构，只有在插件适配、显式汇总、重复复用等必要场景下才展开

### AI 如何既简化又保证写对

需要补充 AI 生成 Flow 的结构规则，避免两个极端：

- 过度压缩：一个块里塞太多职责，结构不清楚
- 过度展开：为每个细碎动作都生成一个块，画布噪音太高

后续需要形成明确约束：

- 用“主任务步骤”而不是“内部微动作”作为默认拆块单位
- 让格式适配优先由系统默认行为承担
- 只有当默认行为不能保证正确性时，才显式增加适配块或模板
- 对 AI 编辑补充“什么时候必须拆块，什么时候不要拆块”的规则

### 7. Flow 创建结构文档化

- 明确 FlowSpec 只用于内部高层创建结构
- 明确 CanonicalFlow 只用于系统内部标准编辑结构
- 明确用户只接触编辑器中的 Flow，而不接触内部结构概念
- 明确 AI 创建和系统收敛的职责边界
