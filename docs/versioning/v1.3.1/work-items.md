# 版本 1.3.1 工作项

## 本次范围

- 收掉 1.3 仍残留的编辑器噪音和内部概念泄漏
- 完成一轮整体检测，确保 FlowSpec 用户层已经真正落到产品表面
- 补齐关键回归与校验，避免 1.3 架构改动只在局部成立

## 本次原则

- 不继续扩张能力边界
- 不把内部引用语法直接当作用户默认心智
- 优先清理“已经能跑但还不够顺”的地方
- 先让默认路径干净，再考虑高级能力

## 准备修改

### 1. 编辑器文案与界面收尾

- 收掉用户默认视角下不该出现的内部实现说明
- 收掉直接暴露 `{{inputs.xxx}}`、`{{steps.xxx}}` 的主路径提示
- 收掉把“稳定引用”“内部字段名”直接强塞给用户的文案
- 继续检查右侧配置面板、运行面板、工具栏和导出入口里的旧语义

目标不是隐藏能力，而是让用户先理解任务步骤，再接触内部引用结构。

### 2. FlowSpec 用户层一致性校准

- 检查所有用户可见的 Flow 入口，确认它们表达的是 FlowSpec 主步骤，而不是 CanonicalFlow 细节
- 检查仍残留的旧块心智、旧命名和旧默认行为
- 明确哪些字段属于用户主操作层，哪些字段应下沉到高级选项或内部层

重点检查：

- Flow 编辑器
- AI 编辑
- 模板制造
- Flow 描述与导出文案

### 3. 默认引用路径继续收口

- 检查是否还有通过自由文本直接操作内部引用的地方
- 继续把“引用上游”收成更偏选择式的交互，而不是模板语法教学
- 明确哪些引用能力仍然留在高级路径

这一项的目标不是彻底删除模板能力，而是避免默认路径继续围绕模板语法组织。

### 4. 插件 / 工具步骤配置补强

- 检查工具步骤字段映射是否已经覆盖默认使用路径
- 补充缺失的 schema 驱动提示、必填提示和映射校验
- 检查复杂输入重组是否仍然会把用户推回内部模板心智

### 5. 整体检测与回归

- 完成一轮覆盖主路径的系统回归
- 明确哪些是自动验证，哪些是手工 smoke test
- 确认以下链路在当前语义下都成立：
  - 创建 Flow
  - AI 编辑 Flow
  - 保存与再次打开
  - 运行与查看中间步骤
  - 工具步骤映射
  - 导出 Flow

### 6. 架构遗漏检查

- 检查是否还有“文档说用户操作 FlowSpec，但实现里用户仍在直接碰内部细节”的断点
- 检查后端导出、代码生成、Agent 描述等旁支路径是否仍残留旧语义
- 检查 FlowSpec、CanonicalFlow、ExecutionPlan 三层之间是否还有职责混淆

这一步的输出应是明确的问题清单，而不是继续含糊地说“后面再看”。

## 建议验收清单

1. 新建一个 Flow，只通过默认 UI 就能完成基本步骤搭建，不需要理解内部模板语法。
2. 在处理步骤中引用上游内容时，用户主要通过选择式交互完成，而不是靠记忆内部引用格式。
3. 结果步骤在主界面里看起来是“终点步骤”，而不是一段需要额外解释的技术语义。
4. 工具步骤的默认配置路径不要求用户先理解复杂 JSON 重组。
5. 从创建、编辑、运行到导出，主要文案和概念一致，不再在不同界面切换心智。

## 与 1.4 的边界

以下事项不放进 1.3.1：

- Flow 测试计划
- 基于中间步骤输出的 AI 测试裁决
- 测试结果复跑与对比

这些属于明确的新能力，应进入 1.4。
