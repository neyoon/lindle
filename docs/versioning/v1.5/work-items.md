# 版本 1.5 工作项

## 本次范围

- 打磨现有 Flow，而不是引入新的 Flow Outline 层
- 保持当前 Workflow / FlowSpec / CanonicalFlow / ExecutionPlan 兼容链路
- 把复杂配置向高级选项收拢
- 把校验结果变成清晰状态和修复引导

## 本次原则

- 用户默认理解 Flow 和 Agent，不需要理解 Tool、Plugin、Schema 或 function calling
- 输入是用户意图，不是表单工程
- 字段是系统执行细节，不是默认建模入口
- 健康检查是视觉状态，不是错误列表
- Flow 打磨优先保证现有工作不被 AI 编辑和保存链路破坏

## 准备修改

### 1. FlowSpec 往返保真

当前 AI 编辑链路会经过 FlowSpec，再物化回 Workflow。

1.5 必须先保证这个往返过程不丢配置。

需要保留：

- 工具步骤的字段映射
- 工具步骤的高级模板
- AI 步骤的模型选择
- 手动连线的输出字段选择
- 输出结构定义
- 已有稳定引用和来源 ID

这项是 1.5 的基础工作。否则用户配置好的 Flow 可能在 AI 编辑后被悄悄改坏。

### 2. 输入体验简化

输入字段配置应从默认路径中后退。

默认状态下，用户只需要知道这个 Flow 需要什么输入材料。

建议的默认体验：

- 无字段或单字段 Flow 显示一个简单输入框
- 多字段 Flow 才展示表单
- Collect 步骤默认显示输入说明
- 字段名、字段类型、必填、默认值进入高级区域
- AI 创建 Flow 时可以自动推断字段，但不要求用户理解字段模型

高级字段配置仍然保留，因为它对 Agent 调用 Flow、Skill 生成和结构化执行有价值。

### 3. Flow 健康检查视觉化

健康检查不能只是保存或运行时报错。

它应该成为 Flow 编辑器中的清晰状态。

状态建议分为：

- 可运行
- 需要补充
- 有阻塞问题
- 运行后发现问题

视觉表现应明确：

- 工具栏显示整体状态
- 有问题的步骤在画布上直接标记
- 右侧配置面板显示当前步骤的问题和修复入口
- 运行按钮在阻塞状态下提示需要先补什么

健康检查不应制造新的复杂表单。

它只回答三个问题：

- 当前 Flow 能不能运行
- 哪一步需要处理
- 用户下一步应该点哪里修

### 4. 执行反馈增强

当前运行事件已经能显示步骤开始、完成和最终结果。

1.5 需要让失败定位更明确。

建议增强：

- 增加步骤级失败状态
- 失败时保留对应步骤 ID 和步骤名称
- 每个步骤显示耗时、输入摘要和输出预览
- 最终输出默认展示，步骤详情默认可折叠
- 画布块状态与运行面板保持同步

执行反馈的目标不是展示更多日志，而是让用户知道问题发生在哪里。

### 5. 工具字段映射打磨

工具步骤的默认输入语义已经是结构化上游结果。

1.5 应继续强化字段映射，减少用户手写模板。

建议增强：

- 根据工具输入要求显示字段完成度
- 必填字段未映射时进入健康检查
- 新增工具步骤时自动生成待配置映射项
- 优先引导用户选择变量或常量
- 高级模板只在字段映射无法表达时展开

工具步骤应让普通用户感觉是在“选择输入来源”，而不是在写 JSON。

### 6. 保存和运行前修复引导

保存和运行前的校验需要保持轻量。

建议行为：

- 可自动修复的问题由系统修复
- 阻塞问题才阻止运行
- 非阻塞问题以提示方式显示
- 每个提示都应指向具体步骤
- 用户可以从提示直接打开对应配置面板

这里的重点是减少挫败感，而不是提高报错覆盖率。

## 不在本次范围

- 不引入 Flow Outline 作为默认创建层
- 不重做整个画布结构
- 不把字段配置扩展成完整表单系统
- 不把健康检查做成复杂测试系统
- 不做批量回归测试和 AI 自动修复

这些内容放到后续版本继续演化。
